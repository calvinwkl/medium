Xcode Starter Guide (3) - Common Practices to Start an iOS App with CocoaPods

In the previous tutorials, you have learnt how to set up your Xcode and start your very first running iOS app. To recap, here are the links to Xcode Starter Guide (1) — Learn How To Use Xcode and Xcode Starter Guide (2) — Run Your First App.

Now I will introduce CocoaPods. A dependency manager that helps you to acquire others’ code for your own project use. So what is a dependency and which are the dependencies you have to manage?

Before that, You know you have your basic tools perfectly set up. You also now know how to start coding. It sounds pretty fine but the thing is, in the programming world, there is a lot of codes already written by other developers. Thanks to the open source communities, you can opt to use their code and simply enjoy the fruits of others’ work.

Each of others’ project is called a dependency. If you need networking library, you probably want to use a dependency called Alamofire. If you want to display image, you might need SDWebImage’s help. There are many well established projects in the community. You definitely want a tools to manage them. So here comes CocoaPods.

Using CocoaPods or Carthage is a must to manage 3rd party libraries to make great apps

There actually is another dependency manager called Carthage. Although CocoaPods is more popular and supported all of the dependencies, Carthage has its own advantages and is preferred by some developers. All in all, you definitely want to go with either one. But in this tutorial, I will only walk you through CocoaPods.

The source code of this tutorial is available on Github.

CocoaPods Setup

First you have to install CocoaPods in your command line first.

sudo gem install cocoapods

After that, you navigate to your Xcode project root directory.

cd /Users/${YOUR_USER_NAME)}/${YOUR_XCODE_ROOT}

So in my case it will be like below, the root directory is the directory containing your .xcodeproj file.

cd /Users/calvin/xcode-starter
// contains starter.xcodeproj file

And then you initialize Cocoapods as below.

pod init

Then you should see a Podfile is created in the directory. Use any editor to open it and you should see the content is similar to below.

The Podfile automatically creates 2 targets. It enables us to add corresponding dependency specifically to each target. You might not have the Tests as mine.

Do not modify the Podfile, then we install the CocoaPods as below.

pod install

You you will see some similar console log like this.

Pods is the name we call the dependencies. We have not changed the default Podfile, so it is correct that no pods is installed.

One simple but big thing is from now on you should only use .xcworkspace to run your project. So do it as instructed in the console log.

Check your root directory, you should see a .xcworkspace is generated. You also see Pods directory and Podfile.lock generated which you do not need to worry about or modify it. It will all be handled by CocoaPods.

So open your project with .xcworkspace, hit run in the simulator, it should be working as normal. If one day you suddenly cannot run your project successfully as usual, it is probably you mistakenly opened your project with .xcodeproj.

Add a Pod

We can try to add a pod now. In the Podfile inside your app target, add Alamofire as a dependency.

Hit pod install again. You should clearly see Alamofire is installed successfully to the project.

To verify this, re-open .xcworkspace and run again. It should build successfully as intented.

Cool! Alamofire is now in your project. But before we use it. One thing worth mentioning is actually we should specfic pods version in our Podfile.

pod 'Alamofire'

to become

pod 'Alamofire' ~> '4.7.0'

By doing this it will ensure we are downloading the exactly same 4.7.0 version every time. If we do not specify the version. CocoaPods will automatically get the newest version of the Pods. Getting the latest version sounds great but in the real world development, stability is also one extremely important part to take care of.

Specify your Pod version

For example, someday you or you colleagues pod install again and get a 4.7.1 version that will break your app silently. It will be hard to trace any dependency problem if you do not specify the Pods you use.

Use the Pod installed

Open your .xcworkspace again, using Alamofire as an example, in UIViewController you can import this Pods.

import UIKit
import Alamofire

If you can build and run with no error, you are good to go with this Pods.

Conclusion

Cocoapods is a great tools for developers to manage iOS 3rd party libraries in their own projects and a dependency manager (CocoaPods or Carthage in iOS) almost is a must for every nowadays projects. You do not want to build an app without using it.

Whenever you need any Pods, just state it in the Podfile, hit pod install, and import it in the file header in your desired locations and you can use it.

You can search many other Pods on https://cocoapods.org. Go search something and explore it!

This is my end of Xcode Starter Guide series. Hope you find these tutorials useful to start your iOS app development.
